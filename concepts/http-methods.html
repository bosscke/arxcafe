HTTP Methods (Verbs)

HTTP Methods (also called HTTP verbs) define the action to be performed on a resource. They tell the server what operation the client wants to execute: reading data (GET), creating new resources (POST), updating existing ones (PUT), or deleting them (DELETE). These methods form the foundation of RESTful API design and enable standardized communication between clients and servers.

Core HTTP Methods

GET

Retrieve data from the server. Most common HTTP method for reading resources.

  • Read-only operation (safe)
  • Idempotent (same result every time)
  • Can be cached by browsers
  • Parameters in URL query string
  • No request body
  • Visible in browser history
  • Can be bookmarked
  • Length restrictions on URL

POST

Submit data to create a new resource on the server.

  • Creates new resources
  • Not idempotent (repeated calls create duplicates)
  • Not cached by default
  • Data sent in request body
  • Can send large amounts of data
  • Not visible in URL
  • Cannot be bookmarked
  • More secure for sensitive data

PUT

Update or replace an entire existing resource on the server.

  • Updates entire resource
  • Idempotent (same result when repeated)
  • Not cached
  • Data sent in request body
  • Requires full resource representation
  • Creates resource if doesn't exist
  • Overwrites existing data
  • Used in RESTful APIs

DELETE

Remove a resource from the server.

  • Deletes specified resource
  • Idempotent (deleting again has no effect)
  • Not cached
  • Usually no request body
  • Resource identified in URL
  • May return deleted resource in response
  • Permanent operation (typically)
  • Requires proper authorization

Additional HTTP Methods

PATCH

Partially update a resource without replacing the entire object.

  • Updates specific fields only
  • More efficient than PUT for small changes
  • Not always idempotent
  • Sends only changed fields

HEAD

Same as GET but returns only headers, no response body.

  • Check if resource exists
  • Get metadata (size, modified date)
  • Test links before downloading
  • Faster than GET

OPTIONS

Describe communication options for the target resource.

  • Returns allowed HTTP methods
  • Used in CORS preflight requests
  • Discovers server capabilities
  • Safe and idempotent

TRACE

Echoes back the received request for debugging purposes.

  • Diagnostic tool
  • Often disabled for security
  • Shows request path through proxies
  • Rarely used in production

CRUD Operations Mapping

HTTP Methods → Database Operations
POST → CREATE
GET → READ
PUT/PATCH → UPDATE
DELETE → DELETE

RESTful API Example:

GET /api/users → List all users

GET /api/users/123 → Get user with ID 123

POST /api/users → Create new user

PUT /api/users/123 → Replace user 123 entirely

PATCH /api/users/123 → Update specific fields of user 123

DELETE /api/users/123 → Delete user 123

Key Properties

Safe Methods

Do not modify server state. Read-only operations.

  • GET: Retrieve data only
  • HEAD: Get headers only
  • OPTIONS: Query capabilities
  • Can be cached aggressively
  • No side effects on server

Idempotent Methods

Multiple identical requests have the same effect as one request.

  • GET: Always returns same data
  • PUT: Same update = same result
  • DELETE: Deleting deleted resource = no change
  • HEAD, OPTIONS: Always safe and idempotent
  • POST: NOT idempotent (creates duplicates)

GET vs POST Comparison

When to Use GET

  • Retrieving data (search, filters)
  • Shareable/bookmarkable URLs
  • Cacheable requests
  • Non-sensitive data
  • Read-only operations
  • Browser back/forward navigation
  • Small amounts of data

When to Use POST

  • Creating new resources
  • Submitting forms
  • Sensitive data (passwords)
  • Large data payloads
  • File uploads
  • Non-idempotent operations
  • Actions with side effects

Common Mistakes

HTTP Method Anti-Patterns

  • Using GET for Mutations: Never use GET to create, update, or delete data
  • Sensitive Data in GET: Don't put passwords or tokens in URL parameters
  • Not Using Idempotent Methods: PUT and DELETE should be safely repeatable
  • POST for Everything: Don't default to POST when GET, PUT, or DELETE is appropriate
  • Ignoring HTTP Semantics: Use methods according to their intended purpose
  • Wrong Status Codes: Return correct codes (200 OK, 201 Created, 204 No Content)
  • PUT Without Full Resource: PUT should send complete representation
  • DELETE With Body: Avoid sending request body with DELETE

Best Practices

Using HTTP Methods Correctly:

  • Follow REST Principles: Use standard CRUD mapping (POST=Create, GET=Read, PUT=Update, DELETE=Delete)
  • Use Correct Status Codes: 200 OK, 201 Created, 204 No Content, 404 Not Found, etc.
  • Implement Idempotency: Ensure PUT, DELETE, GET produce same result when repeated
  • Version Your APIs: Use /v1/, /v2/ prefixes to maintain backward compatibility
  • Validate Input: Always validate request data on server-side
  • Use PATCH for Partial Updates: More efficient than replacing entire resource with PUT
  • Document API Endpoints: Clear documentation of methods, parameters, responses
  • Handle OPTIONS for CORS: Support preflight requests properly
  • Secure Sensitive Operations: Require authentication for POST, PUT, DELETE
  • Return Appropriate Data: POST/PUT return created/updated resource, DELETE returns 204 or deleted item

JavaScript Fetch API Examples:

HTTP methods are the verbs of the web, defining what action to perform on resources. Understanding and correctly implementing these methods is essential for building RESTful APIs and creating robust, maintainable web applications that follow web standards.
