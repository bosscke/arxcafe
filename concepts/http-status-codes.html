HTTP Status Codes

HTTP Status Codes are three-digit numbers returned by servers in HTTP responses to indicate the result of a client's request. They provide standardized feedback about whether the request succeeded, failed, was redirected, or encountered an error. Status codes are grouped into five classes (1xx-5xx), each representing a different category of response, helping developers and systems handle outcomes appropriately.

Status Code Categories

Five Status Code Classes
1xx Informational
2xx Success
3xx Redirection
4xx Client Error
5xx Server Error

2xx Success Codes

Request Succeeded

  • 200 OK: Standard success response, request processed successfully
  • 201 Created: Resource successfully created (POST requests)
  • 202 Accepted: Request accepted for processing, not yet completed
  • 203 Non-Authoritative Information: Modified version of 200 from proxy
  • 204 No Content: Success but no content to return (DELETE, PUT)
  • 205 Reset Content: Client should reset document view
  • 206 Partial Content: Partial resource delivered (range requests)

3xx Redirection Codes

Further Action Required

  • 300 Multiple Choices: Multiple possible responses, user must choose
  • 301 Moved Permanently: Resource permanently moved to new URL
  • 302 Found: Resource temporarily at different URL
  • 303 See Other: Response found at another URL via GET
  • 304 Not Modified: Resource unchanged, use cached version
  • 307 Temporary Redirect: Same as 302 but preserve HTTP method
  • 308 Permanent Redirect: Same as 301 but preserve HTTP method

4xx Client Error Codes

Common Client Errors

  • 400 Bad Request: Malformed request syntax
  • 401 Unauthorized: Authentication required
  • 403 Forbidden: No permission to access
  • 404 Not Found: Resource doesn't exist
  • 405 Method Not Allowed: HTTP method not supported
  • 408 Request Timeout: Server timed out waiting

Advanced Client Errors

  • 409 Conflict: Request conflicts with current state
  • 410 Gone: Resource permanently removed
  • 413 Payload Too Large: Request body too large
  • 415 Unsupported Media Type: Content-Type not supported
  • 429 Too Many Requests: Rate limit exceeded
  • 451 Unavailable For Legal Reasons: Censored

5xx Server Error Codes

Server-Side Failures

  • 500 Internal Server Error: Generic server error, something went wrong
  • 501 Not Implemented: Server doesn't support functionality
  • 502 Bad Gateway: Invalid response from upstream server
  • 503 Service Unavailable: Server overloaded or under maintenance
  • 504 Gateway Timeout: Upstream server didn't respond in time
  • 505 HTTP Version Not Supported: HTTP version not supported
  • 507 Insufficient Storage: Server out of storage space
  • 511 Network Authentication Required: Client must authenticate to network

1xx Informational Codes

Interim Responses

  • 100 Continue: Client should continue with request (used with Expect header)
  • 101 Switching Protocols: Server switching to protocol specified by client
  • 102 Processing: Server received and is processing, but no response yet (WebDAV)
  • 103 Early Hints: Return some response headers before final response

Real-World Scenarios

Successful Operations

200: Page loads successfully, API returns data

201: User account created, blog post published

204: Resource deleted successfully, settings updated

Redirects in Practice

301: Old URL → New URL permanently (SEO-friendly)

302: Temporary maintenance redirect

304: Browser uses cached CSS, images, fonts

User Errors

400: Invalid JSON in API request

401: Login required to access dashboard

403: Admin-only page accessed by regular user

404: Page doesn't exist, typo in URL

429: API rate limit hit

Server Issues

500: Unhandled exception, database crash

502: Reverse proxy can't reach backend

503: Server restarting, too much traffic

504: Backend service timeout

Common Use Cases by Method

Status Codes for Different HTTP Methods:

GET Request:

  • ✓ 200 OK - Resource found and returned
  • ✓ 304 Not Modified - Use cached version
  • ✗ 404 Not Found - Resource doesn't exist

POST Request:

  • ✓ 201 Created - New resource created successfully
  • ✓ 200 OK - Action completed successfully
  • ✗ 400 Bad Request - Invalid input data
  • ✗ 409 Conflict - Resource already exists

PUT/PATCH Request:

  • ✓ 200 OK - Updated successfully, returns resource
  • ✓ 204 No Content - Updated successfully, no body
  • ✗ 404 Not Found - Resource to update doesn't exist
  • ✗ 422 Unprocessable Entity - Validation failed

DELETE Request:

  • ✓ 204 No Content - Deleted successfully
  • ✓ 200 OK - Deleted, returns deleted resource
  • ✗ 404 Not Found - Resource already deleted or doesn't exist

Best Practices

Using Status Codes Correctly:

  • Return Accurate Codes: Use the most specific appropriate code for the situation
  • Be Consistent: Use the same codes for similar operations across your API
  • Include Error Messages: Provide helpful error descriptions in response body
  • Document Your Codes: Clearly specify which codes your API returns
  • Don't Overuse 200: Use 201, 204, etc. when more appropriate
  • Proper 4xx vs 5xx: Client errors are 4xx, server errors are 5xx
  • Use 404 Wisely: Don't expose sensitive information about what exists
  • Implement Rate Limiting: Return 429 when limits exceeded
  • Maintenance Mode: Use 503 with Retry-After header during downtime
  • Log Server Errors: Always log 5xx errors for debugging

Code Examples

Handling Status Codes in JavaScript:

// Fetch with status code handling fetch('/api/users/123') .then(response => { if (response.status === 200) { return response.json(); } else if (response.status === 404) { throw new Error('User not found'); } else if (response.status === 401) { throw new Error('Please log in'); } else if (response.status >= 500) { throw new Error('Server error, please try again'); } throw new Error(`Unexpected status: ${response.status}`); }) .then(data => console.log(data)) .catch(error => console.error(error)); // Using response.ok (true for 200-299) fetch('/api/data') .then(response => { if (!response.ok) { throw new Error(`HTTP ${response.status}: ${response.statusText}`); } return response.json(); }); // Express.js server returning status codes app.post('/api/users', (req, res) => { const user = createUser(req.body); res.status(201).json(user); // 201 Created }); app.delete('/api/users/:id', (req, res) => { deleteUser(req.params.id); res.status(204).send(); // 204 No Content }); app.get('/api/users/:id', (req, res) => { const user = findUser(req.params.id); if (!user) { return res.status(404).json({ error: 'User not found' }); } res.status(200).json(user); });

Common Mistakes

Status Code Anti-Patterns

  • Always Returning 200: Even for errors (breaks HTTP semantics)
  • Using 500 for Client Errors: Use 4xx for client mistakes
  • Not Including Error Details: Status code alone isn't enough
  • Inconsistent Codes: Same operation returns different codes
  • Exposing Internal Errors: Don't leak stack traces in 500 responses
  • Wrong Redirect Codes: Using 302 when 301 is appropriate (SEO impact)
  • Not Implementing 429: No rate limiting protection
  • Ignoring 304: Not supporting conditional requests (wastes bandwidth)

HTTP Status Codes are the language servers use to communicate outcomes. Proper use of status codes creates predictable, debuggable APIs, enables better error handling, improves caching, and provides clear feedback to clients about what happened with their requests.
