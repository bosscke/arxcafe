Pagination

Pagination lets APIs return large datasets in manageable chunks. Good pagination is predictable, performant, and safe from abuse (e.g., unbounded queries).

Approaches

Offset / Limit

Simple: ?offset=0&limit=50. Risk: slow on deep pages; duplicates if data shifts.

Cursor

Stable ordering + cursor token (e.g., encoded last ID). Better for real-time lists.

Page / Size

UI-friendly ?page=2&size=25; internally similar to offset.

Keyset

Use indexed sort key (e.g., created_at, id) for fast WHERE key > lastKey.

Examples

Offset Pagination (SQL)

SELECT * FROM orders ORDER BY created_at DESC OFFSET 100 LIMIT 20;

Keyset/Cursor (SQL)

-- given last_created_at from previous page tSELECT * FROM orders WHERE created_at < :last_created_at ORDER BY created_at DESC LIMIT 20;

API Response Shape (Cursor)

{ "data": [{"id": "o1"}, {"id": "o2"}], "page": { "limit": 20, "nextCursor": "eyJsYXN0SWQiOiJvMiJ9", "hasMore": true } }

Express Handler (Cursor)

const express = require('express'); const app = express(); app.get('/orders', async (req,res)=>{ const limit = Math.min(parseInt(req.query.limit||'20',10), 100); const cursor = req.query.cursor || null; // decode to lastId/createdAt const page = await findOrders({ limit, cursor }); res.json({ data: page.items, page: { limit, nextCursor: page.nextCursor, hasMore: page.hasMore } }); }); app.listen(3000);

Tips

Limits

Enforce max page size; default sensible limits (e.g., 20â€“100).

Sorting

Stable, indexed sort key; avoid non-deterministic orders.

Counts

Full counts are expensive; consider approximate or separate endpoints.

Caching

Cache safe GETs; vary by cursor/offset params.

Common Pitfalls

Avoid these:

  • Unbounded queries (no limit)
  • Unstable sort keys causing duplicates/gaps
  • Deep offset pagination on large tables without indexes
  • Returning massive total counts on every request

Choose offset for simplicity, cursor/keyset for correctness and scale. Always cap limits, use stable ordering, and design predictable responses.
