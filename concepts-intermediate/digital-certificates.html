Digital Certificates

Digital certificates (X.509) bind a public key to an identity (domain, org, person). They are cryptographically signed by a Certificate Authority (CA), enabling trust and authentication in TLS, code signing, email encryption, and more.

Certificate Anatomy

Subject & Issuer

  • Subject: who/what the cert identifies (domain, org)
  • Issuer: the CA that signed it
  • Chain of trust to root CA

Public Key

  • RSA (2048/4096) or ECDSA (P-256/384)
  • Used by clients to verify signatures or encrypt
  • Private key never leaves server

Validity Period

  • Not Before / Not After timestamps
  • Modern certs limited to 398 days
  • Auto-renewal (ACME/Let's Encrypt) critical

Extensions

  • SAN (Subject Alternative Names): multi-domain
  • Key Usage: digital signature, encryption, etc.
  • EKU (Extended Key Usage): server auth, code signing

Chain of Trust

How browsers validate a certificate:
Server presents cert + intermediate chain
Client checks signature against issuer
Chain walks up to root CA in trust store
Root CA is pre-trusted by OS/browser
Cert validates: hostname, dates, revocation
Handshake continues using public key

Certificate Types

DV (Domain Validated)

Proves domain control only. Quick, cheap, automated (ACME/Let's Encrypt). No org verification.

OV (Organization Validated)

Verifies legal entity. Shows org name in cert. Moderate cost, manual vetting.

EV (Extended Validation)

Strict identity checks. Browser shows org name (historically in address bar). Costly, thorough.

Wildcard & Multi-Domain

Wildcard (*.example.com) covers subdomains. SAN certs list multiple names.

Revocation Mechanisms

CRL (Certificate Revocation List)

  • CA publishes list of revoked cert serials
  • Clients download/check periodically
  • Slow, heavy for large lists

OCSP (Online Certificate Status Protocol)

  • Real-time revocation check per cert
  • Privacy risk: CA sees what you visit
  • Latency on every handshake

OCSP Stapling

  • Server fetches signed OCSP response
  • Includes in TLS handshake
  • Faster, more private

CRLSets & Must-Staple

  • Browser-specific revocation sets
  • Must-Staple extension forces OCSP stapling
  • Better protection against compromise

Tools & Examples

OpenSSL: view certificate

$ openssl s_client -connect example.com:443 -servername example.com < /dev/null | openssl x509 -text -noout

Let's Encrypt (Certbot) renewal

$ certbot renew --dry-run $ certbot renew

Windows: view local cert stores (PowerShell)

PS> Get-ChildItem -Path Cert:\LocalMachine\Root PS> Get-ChildItem -Path Cert:\CurrentUser\My

Best Practices

Secure certificate management:

  • Auto-renew with ACME (Let's Encrypt, ZeroSSL)
  • Include full chain (leaf + intermediates) in server config
  • Enable OCSP stapling for speed and privacy
  • Use strong keys: RSA 2048+ or ECDSA P-256+
  • Monitor expiry; set alerts 30+ days before
  • Keep private keys offline/encrypted; rotate if compromised
  • Validate hostname matches (CN/SAN) before deployment

Common Pitfalls

Watch out for:

  • Expired certificates (monitor and auto-renew)
  • Hostname mismatch (CN/SAN != requested domain)
  • Missing intermediate certs in chain
  • Self-signed or untrusted CA in production
  • Weak key sizes (RSA < 2048, MD5/SHA1 signatures)
  • Private key exposure or reuse across envs

Digital certificates underpin secure communication on the web. Proper issuance, chain validation, timely renewal, and revocation handling keep trust intact and attacks at bay.
