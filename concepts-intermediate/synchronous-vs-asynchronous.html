Synchronous vs Asynchronous Communication

Synchronous communication is request/response where the caller waits for a result. Asynchronous communication is message or event-driven where the caller does not wait; work continues and results arrive later. Good systems use both—choosing sync for immediacy and async for resilience and decoupling.

Comparing Approaches

Synchronous

  • Immediate response
  • Simple error handling
  • Tighter coupling and higher latency risk

Asynchronous

  • Decoupled producers/consumers
  • Resilient to spikes, enables retries
  • Eventual consistency and more complex flow

When to Choose

  • Sync: user actions needing instant feedback
  • Async: long-running tasks, integrations, pipelines
  • Hybrid: sync accept, async processing

Examples

Sync REST (Node.js)

// service-a/server.js const express = require('express'); const fetch = require('node-fetch'); const app = express(); app.get('/profile/:id', async (req,res)=>{ const controller = new AbortController(); const timeout = setTimeout(()=>controller.abort(), 2000); // 2s timeout try { const r = await fetch(`http://service-b:4001/users/${req.params.id}`, { signal: controller.signal }); clearTimeout(timeout); if (!r.ok) return res.status(502).json({ error: 'upstream error' }); return res.json(await r.json()); } catch (e) { return res.status(504).json({ error: 'timeout or network' }); } }); app.listen(4000);

Async Event (Kafka-like)

// publisher.js const bus = require('./bus'); function placeOrder(dto){ bus.publish('order.placed', { id: 'o-123', userId: dto.userId, items: dto.items }); return { accepted: true }; } // consumer.js const bus = require('./bus'); bus.subscribe('order.placed', async (evt)=>{ // process asynchronously, retry on failure try { await chargePayment(evt.id); await reserveStock(evt.items); } catch (e) { // send to dead-letter or schedule retry } });

Simple In-Process Bus

// bus.js const { EventEmitter } = require('events'); const bus = new EventEmitter(); module.exports = { publish: (type, payload) => bus.emit(type, payload), subscribe: (type, handler) => bus.on(type, handler) };

Patterns & Considerations

Timeouts & Retries

Always set timeouts; use exponential backoff and limit retries.

Circuit Breakers

Trip on failures to protect dependencies; recover when healthy.

Idempotency

Make operations safe to retry; use idempotency keys.

Delivery Semantics

At-least-once vs exactly-once; deduplicate at consumers.

Best Practices

Do this:

  • Use sync for short, user-facing operations
  • Buffer and queue work for async pipelines
  • Propagate correlation IDs across calls/events
  • Monitor latencies, queue depths, and failure rates
  • Document contracts and delivery guarantees

Common Pitfalls

Avoid these:

  • Blocking synchronous chains causing cascading latency
  • Fire-and-forget without dead-letter handling
  • Mixing business logic into messaging infrastructure
  • Assuming exactly-once delivery without safeguards

Choose synchronous for immediacy, asynchronous for scale and resilience. Many teams accept requests synchronously, then process asynchronously—getting the best of both worlds.
