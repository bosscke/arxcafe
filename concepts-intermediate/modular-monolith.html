Modular Monolith

A Modular Monolith is a single deployable application split into well-defined modules with strict boundaries and internal APIs. It keeps the operational simplicity of a monolith while improving maintainability, testability, and team autonomy—often serving as a clean stepping stone to microservices later.

Why Choose Modular Monolith

Clear Boundaries

  • Modules own their domain and APIs
  • Encapsulation reduces coupling
  • Fewer cross-cutting side effects

Operational Simplicity

  • Single build, single deploy
  • No inter-service networking
  • Straightforward observability

Evolution Path

  • Modules can be extracted later
  • Aligns with DDD bounded contexts
  • Minimizes “distributed monolith” risk

Module Boundaries & Communication

Internal APIs

Expose module facades (services/interfaces). Other modules depend on interfaces, not internals.

Dependencies

One-way dependencies. Avoid cyclical imports. Keep shared utilities minimal and stable.

Data Access

Each module owns its data-access layer. No direct cross-module DB queries.

Events

Publish in-process domain events to decouple features without networking overhead.

Example Structure

Folders & Facades

app/ modules/ users/ api/UserService.js domain/User.js infra/UserRepository.js orders/ api/OrderService.js domain/Order.js infra/OrderRepository.js shared/ events/EventBus.js utils/Result.js

Module API Example (Node.js)

// users/api/UserService.js class UserService { constructor(userRepo) { this.userRepo = userRepo; } async getById(id) { return await this.userRepo.findById(id); } } module.exports = UserService; // orders/api/OrderService.js class OrderService { constructor(orderRepo, userService) { this.orderRepo = orderRepo; this.userService = userService; // depend on facade } async placeOrder(dto) { const user = await this.userService.getById(dto.userId); if (!user) throw new Error('User not found'); return await this.orderRepo.create(dto); } } module.exports = OrderService;

Deployment & Scaling

Single Artifact

One container/binary. Modules compile and ship together.

Scale Vertically or Replicate

Increase instance size or run multiple replicas behind a load balancer.

Feature Flags

Enable/disable modules or features at runtime without redeploying separate services.

Observability

Centralized logs/metrics/traces. Tag events by module for visibility.

Best Practices

Keep modules clean:

  • Use DDD to define bounded contexts
  • Expose facades; hide internals
  • One-way dependencies; avoid cycles
  • No cross-module DB queries; go through APIs
  • Automated tests per module + contract tests
  • Architecture linting (imports allowed per module)
  • Plan extraction path (build artifacts, ownership)

Common Pitfalls

Avoid these:

  • Leaking internals (direct imports across modules)
  • Shared DB tables accessed by multiple modules
  • Global state that couples modules
  • Unclear ownership and responsibilities
  • Turning into a distributed monolith prematurely

A Modular Monolith balances simplicity and structure. By enforcing boundaries and internal APIs, you get a maintainable codebase today and an easier path to microservices when—and if—scale demands it.
