Public Key vs Private Key

Public-key cryptography uses pairs of mathematically linked keys: a public key that can be shared openly and a private key kept secret. Data encrypted with one can only be decrypted by the other, enabling secure communication and digital signatures without pre-shared secrets.

Key Roles

Public Key

  • Shared freely (certs, key servers, repos)
  • Used to encrypt messages to you
  • Used to verify signatures you create
  • Cannot decrypt or sign on its own

Private Key

  • Kept secret, never shared
  • Used to decrypt messages encrypted to you
  • Used to sign data to prove identity
  • Compromise means full identity/data breach

How It Works

Encryption (confidentiality):
Alice encrypts with Bob's public key
Message travels over untrusted network
Only Bob's private key can decrypt
Bob reads plaintext message
Digital Signature (authenticity & integrity):
Alice signs message with her private key
Anyone can verify with Alice's public key
Proves Alice authored it and it's unmodified
Non-repudiation: Alice can't deny signing

Common Algorithms

RSA

  • Encryption & signatures
  • Key sizes: 2048 (min), 3072, 4096
  • Mature, widely supported
  • Slower than ECC for same security

ECDSA / EdDSA

  • Elliptic Curve signatures
  • Smaller keys, faster operations
  • P-256, P-384, Ed25519, Ed448
  • Modern TLS/SSH/JWT default

Diffie-Hellman (ECDH)

  • Key agreement, not encryption/signing
  • Both sides derive shared secret
  • Ephemeral for forward secrecy (ECDHE)

Hybrid Schemes

  • Use public-key to exchange symmetric key
  • Bulk data encrypted with AES/ChaCha20
  • TLS, PGP, S/MIME all hybrid

Real-World Use Cases

TLS / HTTPS

Server cert contains public key; server uses private key in handshake to prove identity.

SSH Authentication

User's public key on server; private key on client proves identity without passwords.

Code Signing

Developer signs code with private key; users verify with public key from cert.

Email (S/MIME, PGP)

Public key encrypts email; private key decrypts. Signing proves sender authenticity.

Tools & Examples

Generate RSA key pair (OpenSSL)

$ openssl genrsa -out private.pem 2048 $ openssl rsa -in private.pem -pubout -out public.pem

Generate Ed25519 SSH key pair

$ ssh-keygen -t ed25519 -C "user@example.com"

Node.js: sign/verify (crypto module)

const { sign, verify, generateKeyPairSync } = require('crypto'); const { publicKey, privateKey } = generateKeyPairSync('rsa', { modulusLength: 2048 }); const data = Buffer.from('Hello'); const sig = sign('sha256', data, privateKey); const ok = verify('sha256', data, publicKey, sig); console.log(ok); // true

Best Practices

Secure key management:

  • Never share or transmit your private key
  • Store private keys encrypted at rest (passphrase-protected)
  • Use HSM/TPM for high-value keys (servers, CAs)
  • Rotate keys periodically; revoke compromised keys
  • Minimum RSA 2048 bits; prefer ECDSA/EdDSA when possible
  • Separate keys per use case (SSH, signing, encryption)
  • Backup private keys securely (encrypted, offline)

Common Pitfalls

Avoid these mistakes:

  • Committing private keys to git/repos
  • Using weak/short keys (RSA < 2048, DES)
  • Reusing same key across multiple purposes
  • Storing unencrypted keys on disk/cloud
  • Ignoring key expiry/revocation
  • Trusting unverified public keys (MitM risk)

Public-key cryptography enables secure, scalable communication without pre-shared secrets. Protect private keys fiercely, distribute public keys freely, and choose strong algorithms to keep data and identities safe.
