REST APIs

REST (Representational State Transfer) is an architectural style for building web APIs using HTTP verbs, resource-oriented URLs, stateless requests, and standardized status codes. Good REST APIs are predictable, cacheable, and well-documented.

Core Principles

Resources

Model nouns (e.g., /users, /orders), not actions.

HTTP Verbs

Use GET/POST/PUT/PATCH/DELETE consistently for CRUD semantics.

Statelessness

No server session; every request is complete with auth/context.

Cacheability

Use proper headers (ETag, Cache-Control) for safe GET responses.

Example: Node.js REST API

Express Routes

const express = require('express'); const app = express(); app.use(express.json()); app.get('/users', listUsers); // safe, cacheable app.post('/users', createUser); // creates app.get('/users/:id', getUser); // safe, cacheable app.patch('/users/:id', updateUser); // partial update app.delete('/users/:id', deleteUser); // deletes app.listen(3000);

Controller & Validation

async function createUser(req, res){ const { email, name } = req.body; if (!email) return res.status(400).json({ error: 'email required' }); const user = await saveUser({ email, name }); res.status(201).json(user); }

HTTP Status Codes

// examples 200 OK // success 201 Created // new resource 204 No Content // deleted/updated with no body 400 Bad Request // validation 401 Unauthorized// auth needed 403 Forbidden // auth ok, not allowed 404 Not Found // resource missing 409 Conflict // versioning or duplicates 429 Too Many Requests // rate limited 500 Internal Server Error

Design Tips

Versioning

Use URI versioning (e.g., /v1/) or header-based; keep backward compatible.

Filtering & Pagination

Support ?limit, ?offset/?cursor, ?sort, ?filter=.

Errors

Return structured errors with codes, messages, and fields.

Security

Use TLS, OAuth/JWT, rate limits, and input validation. Avoid over-sharing via fields whitelists.

Best Practices

Do this:

  • Consistent naming and plural nouns
  • Use idempotent methods (PUT/PATCH) where applicable
  • Validate input; limit payload sizes
  • Support ETag/If-None-Match for cache-friendly GETs
  • Provide OpenAPI/Swagger docs

Common Pitfalls

Avoid these:

  • Verb-based routes like /createUser
  • Overloading POST for updates/deletes
  • Returning 200 OK for errors
  • Breaking changes without versioning
  • Leaking internal IDs or sensitive fields

REST APIs work best when they are predictable, well-versioned, and secure. Keep resources clean, verbs consistent, and documentation up to date.
