API Versioning

API versioning lets you evolve contracts without breaking existing clients. Choose a strategy (URI, header, content negotiation), deprecate safely, and keep backward compatibility as long as needed.

Strategies

URI Versioning

Prefix paths (e.g., /v1/, /v2/). Easiest to cache and route.

Header Versioning

Custom or Accept header (media types like application/vnd.app.v2+json).

Query Versioning

Use ?version=2. Simple but less cache-friendly.

GraphQL / gRPC

Prefer field-level evolution with deprecations instead of whole-version jumps.

Examples

Express Router per Version

const express = require('express'); const app = express(); const v1 = express.Router(); const v2 = express.Router(); v1.get('/users/:id', (req,res)=> res.json({ id: req.params.id, name: 'Ada', role: 'user' })); v2.get('/users/:id', (req,res)=> res.json({ id: req.params.id, name: 'Ada', roles: ['user'], email: 'ada@example.com' })); app.use('/v1', v1); app.use('/v2', v2); app.listen(3000);

Media-Type Versioning

// Accept: application/vnd.app.v2+json app.get('/users/:id', (req,res)=>{ const accept = req.headers['accept'] || ''; const isV2 = accept.includes('vnd.app.v2+json'); if (isV2) return res.json({ version: 'v2', id: req.params.id }); return res.json({ version: 'v1', id: req.params.id }); });

Deprecation Headers

// RFC 8594 style res.setHeader('Deprecation', 'true'); res.setHeader('Sunset', 'Wed, 30 Apr 2025 23:59:59 GMT'); res.setHeader('Link', '; rel="deprecation"');

Guidance

Semantic Changes

Add fields in new versions; avoid breaking required fields. Use defaults.

Deprecation Policy

Announce timelines, provide migration docs, and monitor usage before removal.

Docs & SDKs

Publish OpenAPI per version; update SDKs and examples together.

Testing

Contract tests per version; monitor error rates and field usage.

Best Practices

Do this:

  • Keep old versions stable; only patch for security/bugs
  • Additive changes where possible (new fields, endpoints)
  • Set deprecation headers and communicate timelines
  • Provide changelogs and migration guides
  • Monitor per-version usage to plan sunsetting

Common Pitfalls

Avoid these:

  • Breaking changes without a new version
  • Silent behavior changes between versions
  • Running too many active versions without a sunset plan
  • Inconsistent docs and SDKs per version

Versioning protects clients and lets your API evolve. Pick a strategy, document it, and manage deprecations transparently.
