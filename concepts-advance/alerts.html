Alerts

Alerts are notifications triggered when monitored metrics breach defined thresholds or when anomalies are detected. Alerts bridge the gap between monitoring data and human action, ensuring teams know immediately when systems require attention. Effective alerting focuses on symptoms that require action, not all anomalies, reducing noise while catching real problems.

Core Concepts

Alert vs Warning

Different urgency levels.

  • Alert: Immediate action required
  • Warning: Investigate but not urgent
  • Alerts page on-call engineers
  • Warnings trigger dashboards
  • Clear severity definition

Alert Components

What makes up an alert.

  • Metric: What is being measured
  • Threshold: When to trigger
  • Duration: How long before alerting
  • Severity: How critical
  • Action: How to respond

Alert Lifecycle

States an alert progresses through.

  • Firing: Condition is true
  • Acknowledged: Engineer is investigating
  • Resolved: Problem fixed
  • Silenced: Temporarily ignored
  • Escalation: Increased urgency

Alert Routing

Getting alerts to right people.

  • Severity-based routing
  • Service/team ownership
  • On-call rotations
  • Escalation policies
  • Multiple channels

Alert Types

Threshold Alerts

Triggered when metric crosses boundary.

  • Most common alert type
  • Simple comparison logic
  • Static threshold values
  • CPU > 80%, Memory > 90%
  • Easy to understand

Anomaly Alerts

Triggered by unusual patterns.

  • ML-based detection
  • Detects deviations from baseline
  • Adapts to normal variations
  • Fewer false positives
  • Requires historical data

Composite Alerts

Multiple conditions combined.

  • AND/OR logic rules
  • Correlate multiple metrics
  • Reduce false positives
  • E.g., CPU high AND requests high
  • More complex to manage

Absence Alerts

Triggered when data stops arriving.

  • No metric update received
  • Indicates scraper/agent failure
  • Service down detection
  • Heartbeat monitoring
  • Prevents silent failures

Alert Severity Levels

Critical

Immediate response required.

  • Service down
  • Data loss risk
  • Security incident
  • Pages on-call immediately
  • Requires 5-minute response

High

Significant impact, act quickly.

  • Performance degradation
  • Increased error rate
  • Resource exhaustion coming
  • Pages primary on-call
  • Respond within 15 minutes

Medium

Investigate within business hours.

  • Non-critical service affected
  • Minor performance issues
  • Warning signs present
  • Email/Slack notification
  • Respond within 1 hour

Low

Information and trends.

  • Informational only
  • Planning purposes
  • Long-term trends
  • Dashboard visibility only
  • No page required

Notification Channels

Real-Time Channels

Immediate notification delivery.

  • Phone call (critical only)
  • SMS/text messages
  • Push notifications
  • Highest priority alerts
  • Instant acknowledgment

Chat/Messaging

Team communication platforms.

  • Slack channels
  • Microsoft Teams
  • Discord servers
  • Visible to entire team
  • Threaded discussion

Email

For lower urgency alerts.

  • Async notification
  • Searchable records
  • Multiple recipients easily
  • Not ideal for critical
  • Easy to ignore/filter

PagerDuty/Incident Management

Specialized alerting platform.

  • On-call rotation management
  • Escalation policies
  • Alert de-duplication
  • Acknowledgment tracking
  • Post-mortem integration

Alert Fatigue Problems

False Positives

Alerts that don't matter.

  • Incorrect threshold values
  • Transient spikes ignored
  • Seasonal variations
  • Team ignores all alerts
  • Dangerous: real alerts missed

Over-Alerting

Too many alerts fired.

  • Monitoring everything
  • Low thresholds
  • Same incident multiple alerts
  • Team becomes numb
  • Burn-out risk

Alert Noise

Alerts unrelated to problems.

  • Network blips
  • Maintenance windows
  • Expected load patterns
  • Degraded signal quality
  • Need intelligent suppression

Silent Failures

Critical issues missed.

  • Too many alerts to notice
  • Missing alert configurations
  • Alert system down
  • Critical threshold too high
  • Worst possible outcome

Best Practices

Alert on Symptoms

  • Alert on user impact
  • High error rates
  • Slow responses
  • Service unavailability
  • Not internal metrics only

Context in Alerts

  • Include runbook links
  • Show relevant metrics
  • Team/owner information
  • Severity and impact
  • Enable fast action

Tune Thresholds

  • Based on historical data
  • Consider normal variation
  • Time-based thresholds
  • Regular tuning cycles
  • Track accuracy metrics

Actionability

  • Every alert needs action
  • Clear next steps
  • Right person/team
  • Runbooks available
  • Silence non-actionable

Alert Management Strategies

De-duplication

Consolidate related alerts.

  • Same incident multiple sources
  • Reduce noise
  • Group by root cause
  • One incident, one page
  • Intelligent correlation

Suppression/Silence

Temporarily disable alerts.

  • Maintenance windows
  • Scheduled deployments
  • Known false positives
  • Prevent page spam
  • Time-limited silence

Escalation Policies

Route alerts appropriately.

  • Primary/secondary on-call
  • Time-based escalation
  • Severity-based routing
  • Team ownership
  • Manager notification

Runbooks

Documentation for response.

  • Step-by-step guides
  • Linked from alert
  • Common solutions
  • Diagnostic commands
  • Escalation procedures

Alert Quality Metrics

Precision

Ratio of valid to total alerts.

  • True positives / all alerts
  • Higher is better
  • Target: > 90%
  • Reduce false positives
  • Improves team trust

Recall

Ratio of caught to total issues.

  • True positives / all problems
  • Higher is better
  • Target: > 95%
  • Catch real problems
  • Avoid silent failures

Real-World Examples

E-Commerce

Critical checkout alerts.

  • Payment processing error rate > 1%
  • Checkout latency p99 > 3s
  • Cart service unavailable
  • Inventory sync delay > 5min
  • Revenue-impacting metrics

SaaS Platform

Per-customer impact alerts.

  • API endpoint down
  • Database response time spikes
  • Data sync failures
  • Job queue depth growing
  • Customer-specific degradation

Mobile App

Client-side issue detection.

  • App crash rate > 0.5%
  • API timeout rate spike
  • Sync service failures
  • Push notification delivery
  • Version-specific issues

Infrastructure

System-level monitoring.

  • Node unavailable
  • Disk space < 10%
  • Network latency > 100ms
  • Package manager down
  • Security rule violations

Implementing Effective Alerting

Step 1: Define What Matters

  • User-visible problems
  • Business impact
  • Team responsibilities
  • SLO alignment
  • Create rubric

Step 2: Create Initial Alerts

  • Start with critical only
  • Use conservative thresholds
  • Test with real data
  • Include context/runbook
  • Route to right team

Step 3: Measure Quality

  • Track precision/recall
  • Count false positives
  • Measure MTTR
  • Survey team satisfaction
  • Identify patterns

Step 4: Iterate & Improve

  • Tune thresholds based on data
  • Adjust severity levels
  • Add missing alerts
  • Remove noisy alerts
  • Update runbooks

Effective alerting is about balance: catch real problems quickly while avoiding alert fatigue that leads to dangerous alert blindness. Alert on user-visible symptoms, provide actionable context, continuously measure quality, and iterate. Remember: too many alerts is often worse than too few.
